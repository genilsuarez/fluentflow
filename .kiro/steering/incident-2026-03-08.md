# Incident Report: CD Deploy Failure (2026-03-08)

## Resumen

El workflow CD Deploy falló debido a artifacts expirados del CI Build. Los artifacts se retienen solo 7 días y el último commit fue hace más de 7 días.

## Cronología

1. **2026-02-28 01:18** - Último commit exitoso con artifacts (`02e80628`)
2. **2026-03-08 01:05** - Cron job de seguridad ejecutado (diario a las 2 AM)
3. **2026-03-08 01:06** - CD Deploy falló: "artifact not found"
4. **2026-03-08 22:40** - Solución aplicada: commit vacío para regenerar artifacts

## Causa Raíz

### Problema Principal
Los artifacts del CI Build expiran después de 7 días (configurado en `.github/workflows/ci-build.yml`):

```yaml
- name: Upload build artifacts
  uses: actions/upload-artifact@v4
  with:
    name: build-files-${{ github.sha }}
    path: dist/
    retention-days: 7  # ← Artifacts expiran después de 7 días
```

### Flujo del Problema

1. Último commit: 2026-02-28
2. Artifact generado: `build-files-02e80628`
3. Retention: 7 días → Expira: 2026-03-07
4. Cron job de seguridad: 2026-03-08 (después de expiración)
5. CD Deploy intenta descargar artifact expirado → Falla

## Solución Aplicada

### Inmediata
```bash
# Generar nuevo commit para trigger CI/CD completo
git commit --allow-empty -m "chore: trigger CI/CD pipelines"
git push
```

Esto regeneró:
- Nuevos artifacts del CI Build
- Nuevo deployment exitoso
- Commit: `3bcf0f06`

### Mejoras Implementadas

#### 1. CD Deploy Mejorado
Agregado mejor manejo de errores y mensajes informativos en `.github/workflows/cd-deploy.yml`:

- Verificación de existencia de artifact
- Validación de descarga
- Verificación de integridad del zip
- Mensajes de error claros con instrucciones de solución

#### 2. Documentación
Creado `.kiro/steering/ci-cd-maintenance.md` con:

## Prevención Futura

### Opción 1: Commits Regulares (Recomendado)
Hacer commits al menos cada 5-6 días para mantener artifacts frescos.

### Opción 2: Aumentar Retention
Modificar `.github/workflows/ci-build.yml`:

```yaml
retention-days: 30  # En lugar de 7
```

**Pros**: Más tiempo antes de expiración
**Contras**: Mayor uso de storage en GitHub

### Opción 3: Rebuild Automático en CD Deploy
Agregar fallback en CD Deploy para rebuild si artifact no existe.

**Pros**: Totalmente automático
**Contras**: Builds más lentos, duplicación de lógica

### Opción 4: Cron Job de Mantenimiento
Agregar workflow que haga commit vacío cada 5 días:

```yaml
name: Maintenance

on:
  schedule:
    - cron: '0 0 */5 * *'  # Cada 5 días

jobs:
  keep-alive:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git commit --allow-empty -m "chore: keep artifacts fresh"
          git push
```

## Recomendación

**Opción 1 (Commits Regulares)** es la mejor porque:
- No requiere cambios en workflows
- Mantiene el proyecto activo
- Es el flujo natural de desarrollo
- Sin costos adicionales de storage

Si el proyecto entra en mantenimiento sin commits frecuentes, considerar **Opción 4 (Cron Job de Mantenimiento)**.

## Lecciones Aprendidas

1. **Artifact retention** es crítico para CD pipelines que dependen de builds previos
2. **Cron jobs de seguridad** pueden disparar deploys que fallan si artifacts expiraron
3. **Mensajes de error claros** son esenciales para troubleshooting rápido
4. **Documentación proactiva** previene confusión futura
